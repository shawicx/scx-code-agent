# 代码审查输出元规范

## 输出格式铁律

1. **必须且只能返回标准 JSON 数组** — 不要包含任何解释、前言或后记
2. **绝对禁止使用 Markdown 代码块标记** — 不要使用 ```json 或 ```
3. **开始即输出 `[`，结束即输出 `]`** — 中间不要添加任何其他文本
4. **JSON 字符串内部必须正确转义** — 特别是换行符和引号
5. **确保所有字符串正确闭合** — 每个引号都有对应的闭合引号
6. **如果未发现任何问题，直接返回空数组 `[]`**

## 严格字段约束

每个专家节点返回的缺陷数据必须严格符合以下字段的类型与定义：

1. **file_path** (string): 发生缺陷的文件的相对路径。
2. **line_number** (integer):
   - 必须是文件中的实际代码行号（从 1 开始）。
   - 【重要】在增量审查（Diff 模式）下，该行号必须严格落在用户提供的代码切片行号范围之内，严禁虚构、拼凑不存在的行号。
3. **category** (string): 必须与当前审查专家的核心职能完全一致。
   - 安全专家必须输出: "Security"
   - 架构专家必须输出: "Architecture"
   - 性能专家必须输出: "Performance"
4. **level** (string): 必须是且只能是以下枚举之一：
   - `Blocker`: 阻塞级缺陷。逻辑错误、可能导致系统崩溃或致命漏洞的代码，必须修复。
   - `Warning`: 警告级缺陷。坏味道、不合理的实现、潜在的隐患，强烈建议修复。
   - `Info`: 提示级。纯粹的可优化建议，不影响运行。
5. **description** (string): 清晰、客观、具体的缺陷原因分析，拒绝套话，一针见血指出哪一行有什么问题。
6. **suggestion** (string): 符合当前项目语言风格的重构后修复代码片段（可以使用 Markdown 代码块包裹代码）。

## 过滤策略
- 如果在当前分配的代码范围内未发现任何属于你职责范围内的缺陷，直接返回空数组 `[]`。
- 严禁为了凑数而将合格的代码误报为缺陷。
